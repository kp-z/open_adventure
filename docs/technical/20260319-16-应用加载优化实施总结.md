**创建日期**: 2026-03-19

## 概述

本次优化旨在解决用户每次刷新页面都会重新进入加载界面的问题，通过路由状态持久化、初始化逻辑优化和 IndexedDB 缓存，显著提升了应用的加载速度和用户体验。

## 实施内容

### Phase 1: 路由状态持久化

**修改文件**: `frontend/src/app/App.tsx`

**实现内容**:
- 使用 `sessionStorage` 保存当前路由
- 页面刷新时自动恢复到上次访问的页面
- 监听 `popstate` 和 `beforeunload` 事件保存路由状态

**关键代码**:
```typescript
useEffect(() => {
  // 恢复上次的路由
  const lastRoute = sessionStorage.getItem('last-route');
  if (lastRoute && lastRoute !== window.location.pathname) {
    console.log('[Route] 恢复上次路由:', lastRoute);
    window.history.replaceState(null, '', lastRoute);
  }
  setIsRestoring(false);

  // 保存当前路由到 sessionStorage
  const saveRoute = () => {
    const currentPath = window.location.pathname;
    sessionStorage.setItem('last-route', currentPath);
    console.log('[Route] 保存当前路由:', currentPath);
  };

  window.addEventListener('popstate', saveRoute);
  window.addEventListener('beforeunload', saveRoute);

  return () => {
    window.removeEventListener('popstate', saveRoute);
    window.removeEventListener('beforeunload', saveRoute);
  };
}, []);
```

### Phase 3: 启用 IndexedDB 缓存

**修改文件**:
- `frontend/src/lib/api/client.ts` - 添加缓存拦截器
- `frontend/src/lib/api/services/agents.ts` - 启用 agents API 缓存
- `frontend/src/lib/api/services/skills.ts` - 启用 skills API 缓存
- `frontend/src/lib/api/services/workflows.ts` - 启用 workflows API 缓存
- `frontend/src/lib/api/services/teams.ts` - 启用 teams API 缓存
- `frontend/src/lib/api/services/executions.ts` - 启用 executions API 缓存

**实现内容**:
1. **API 客户端缓存拦截器**:
   - 为 `RequestConfig` 添加 `cache` 和 `cacheTTL` 参数
   - 仅对 GET 请求启用缓存
   - 缓存命中时立即返回，后台更新缓存（Stale-While-Revalidate 策略）
   - 缓存键格式: `api:{endpoint}:{params}`

2. **各 API 模块启用缓存**:
   - 所有 `list()` 和 `get()` 方法启用缓存
   - 默认 TTL 为 300 秒（5 分钟）
   - 示例:
     ```typescript
     list: (params?: SkillListParams) =>
       apiClient.get<SkillListResponse>('/skills', {
         params,
         cache: true,
         cacheTTL: 300
       }),
     ```

3. **缓存策略**:
   - **首次请求**: 从服务器获取数据，保存到 IndexedDB
   - **后续请求**: 立即返回缓存数据，后台更新缓存
   - **缓存过期**: 自动删除过期数据，重新请求

## 技术细节

### 缓存键设计
```typescript
const cacheKey = `api:${endpoint}:${JSON.stringify(params || {})}`;
```
- 包含 endpoint 和 params，确保不同请求有不同的缓存键
- 使用 JSON.stringify 序列化 params，保证一致性

### 缓存更新策略
```typescript
if (cached) {
  console.log('[Cache] 命中缓存:', endpoint);
  // 后台更新缓存
  this.fetchAndCache<T>(url, fetchConfig, cacheKey, cacheTTL).catch(() => {});
  return cached;
}
```
- 缓存命中时立即返回，不阻塞用户
- 后台异步更新缓存，确保数据新鲜度
- 更新失败不影响用户体验

### 路由恢复时机
```typescript
const [isRestoring, setIsRestoring] = useState(true);

useEffect(() => {
  // 恢复路由
  const lastRoute = sessionStorage.getItem('last-route');
  if (lastRoute && lastRoute !== window.location.pathname) {
    window.history.replaceState(null, '', lastRoute);
  }
  setIsRestoring(false);
}, []);

if (isRestoring) {
  return null; // 避免闪烁
}
```
- 在 React 挂载前恢复路由
- 避免显示默认路由后再跳转的闪烁

## 已知问题

### 1. 缓存失效时机
- **问题**: 用户修改数据后，缓存未立即失效
- **影响**: 用户可能看到旧数据
- **解决方案**:
  - 在 POST/PUT/DELETE 请求后清除相关缓存
  - 或使用更短的 TTL（如 60 秒）

### 2. 缓存大小限制
- **问题**: IndexedDB 有存储限制（通常 50MB-100MB）
- **影响**: 大量数据可能导致缓存失败
- **解决方案**:
  - 仅缓存列表数据，不缓存详情
  - 定期清理过期缓存
  - 提供手动清除缓存功能（Settings 页面）

### 3. 离线支持
- **问题**: Service Worker 仅在生产环境生效
- **影响**: 开发环境无法测试离线功能
- **解决方案**: 修改 `vite.config.ts`，在开发环境也启用 PWA
