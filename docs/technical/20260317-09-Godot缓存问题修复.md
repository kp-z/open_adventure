# Godot WebAssembly 缓存问题修复

**创建日期**: 2026-03-17
**章节编号**: 09
**作者**: Claude Opus 4.6
**状态**: 已完成

## 问题描述

用户报告虽然已经修复了 Godot 游戏的 API 配置（端口和 HTTP 方法），并重新导出了游戏，但在浏览器中测试时**没有发现任何修复效果**。

### 根本原因

浏览器对 Godot 游戏的关键文件使用了强缓存：
- `index.pck` (9.1 MB) - 包含所有游戏脚本的编译字节码
- `index.wasm` (36 MB) - WebAssembly 运行时

这些文件没有被 Service Worker 预缓存，因此浏览器使用 HTTP 缓存策略。由于 iframe 的 src 是静态路径，没有版本号或时间戳，浏览器会一直使用缓存的旧版本文件。

## 解决方案

采用**方案 A + 方案 C 组合**：

### 方案 A：添加版本号到 iframe src（立即生效）

在 iframe src 中添加版本号查询参数，强制浏览器重新加载游戏文件。

**修改文件**: `frontend/src/app/pages/Microverse.tsx`

**关键修改**:
1. 添加 `gameVersion` 状态
2. 在 `checkVersion()` 中设置版本号
3. 修改 iframe src: `/microverse/index.html?v=${gameVersion}`

### 方案 C：配置后端缓存策略（长期优化）

在后端配置中为 `.pck` 和 `.wasm` 文件设置 `Cache-Control` 头。

**修改文件**: `backend/app/main.py`

**关键修改**:
```python
class MicroverseStaticFiles(StaticFiles):
    async def get_response(self, path: str, scope: Scope):
        response = await super().get_response(path, scope)
        # 为 .pck 和 .wasm 文件设置短期缓存（5 分钟）
        if path.endswith(('.pck', '.wasm')):
            response.headers['Cache-Control'] = 'public, max-age=300'
        return response
```

## 技术细节

### 为什么 .pck 文件中搜索不到字符串？

Godot 导出时会将 GDScript 编译为字节码（.gdc），存储在 .pck 文件中。字节码是二进制格式，无法直接用 `strings` 命令搜索源代码中的字符串常量。

要验证 .pck 文件内容，只能：
1. 检查文件修改时间和大小
2. 在浏览器中运行游戏，查看实际行为
3. 使用 Godot 编辑器的 PCK Explorer 工具（如果有）

### 开发建议

1. **开发时使用硬刷新**：`Cmd+Shift+R` 强制重新加载所有资源
2. **测试时使用隐私模式**：避免缓存干扰
3. **每次修改游戏后更新版本号**：确保用户能看到最新版本
4. **监控 Network 标签**：验证文件是否真的重新加载（状态码 200 而不是 304）
