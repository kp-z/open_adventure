# 二进制应用升级方案设计

## 问题分析

当前 Open Adventure 使用二进制分发（PyInstaller 打包），用户面临的升级挑战：
1. 如何知道有新版本？
2. 如何下载新版本？
3. 如何安全地替换旧版本？
4. 如何保留用户数据和配置？
5. 数据库 schema 如何自动迁移？

## 业界常见升级方案

### 方案 1：手动下载替换（最简单）
**适用场景**：小型工具、早期版本
**流程**：
1. 用户访问 GitHub Releases 页面
2. 下载新版本压缩包
3. 停止旧版本
4. 解压新版本到新目录
5. 复制用户数据到新目录
6. 启动新版本

**优点**：
- 实现简单，无需额外开发
- 用户完全控制升级过程
- 不依赖网络连接

**缺点**：
- 用户体验差，步骤繁琐
- 容易出错（忘记备份数据、路径错误等）
- 用户可能不知道有新版本

### 方案 2：内置版本检查 + 手动下载（推荐）
**适用场景**：中小型应用
**流程**：
1. 应用启动时检查 GitHub API 获取最新版本
2. 如果有新版本，在 UI 中显示通知
3. 用户点击通知，打开 GitHub Release 页面
4. 用户手动下载并替换

**优点**：
- 实现相对简单
- 用户知道有新版本
- 升级过程可控

**缺点**：
- 仍需手动操作
- 数据迁移需要用户处理

### 方案 3：内置自动更新（复杂但体验好）
**适用场景**：成熟的商业应用
**流程**：
1. 应用检查更新
2. 后台下载新版本
3. 验证签名和完整性
4. 提示用户重启
5. 重启时自动替换二进制文件
6. 自动迁移数据

**优点**：
- 用户体验最好
- 自动化程度高

**缺点**：
- 实现复杂（需要更新器、签名验证等）
- 需要处理权限问题（替换二进制文件）
- 可能需要代码签名证书

### 方案 4：容器化部署（Docker）
**适用场景**：服务器端应用
**流程**：
1. 发布新 Docker 镜像
2. 用户执行 `docker pull` 更新镜像
3. 重启容器

**优点**：
- 升级简单（一条命令）
- 环境隔离，不影响宿主机
- 数据持久化通过 volume 管理

**缺点**：
- 需要用户安装 Docker
- 不适合桌面应用

## 推荐方案：渐进式升级策略

### 阶段 1：手动升级 + 数据保护（当前实现）

**立即实现**：
1. **标准化用户数据目录**
   - macOS: `~/.open_adventure/`
   - Linux: `~/.open_adventure/`
   - 包含：数据库、配置、日志、前端资源

2. **升级文档**
   - 在 README 中提供详细的升级步骤
   - 在 Release Notes 中说明数据兼容性

3. **数据库自动迁移**
   - 应用启动时自动执行 Alembic 迁移
   - 迁移失败时给出明确提示

**升级步骤**（用户手册）：
```bash
# 1. 停止旧版本
./stop.sh  # 或 Ctrl+C

# 2. 备份用户数据（可选但推荐）
cp -r ~/.open_adventure ~/.open_adventure.backup

# 3. 下载新版本
wget https://github.com/kp-z/open_adventure/releases/download/v0.2.0/open-adventure-v0.2.0-macos-arm64.tar.gz

# 4. 解压到新目录
tar -xzf open-adventure-v0.2.0-macos-arm64.tar.gz
cd open-adventure

# 5. 启动新版本（会自动迁移数据库）
./start.sh

# 6. 验证升级成功后，删除旧版本和备份
```

### 阶段 2：内置版本检查（v0.3.0 实现）

**功能**：
1. 应用启动时检查 GitHub API
2. Dashboard 显示更新通知
3. 点击通知打开 Release 页面

**实现**：
```python
# backend/app/services/update_checker.py
import aiohttp
from packaging import version

async def check_for_updates():
    """检查 GitHub Releases 是否有新版本"""
    current_version = "0.2.0"

    async with aiohttp.ClientSession() as session:
        async with session.get(
            "https://api.github.com/repos/kp-z/open_adventure/releases/latest"
        ) as resp:
            data = await resp.json()
            latest_version = data["tag_name"].lstrip("v")

            if version.parse(latest_version) > version.parse(current_version):
                return {
                    "has_update": True,
                    "latest_version": latest_version,
                    "release_url": data["html_url"],
                    "release_notes": data["body"]
                }

    return {"has_update": False}
```

**前端显示**：
- Dashboard 顶部显示更新横幅
- 显示版本号和主要更新内容
- 提供"查看详情"和"稍后提醒"按钮

### 阶段 3：一键升级脚本（v0.4.0 实现）

**功能**：
提供升级脚本，自动化升级过程

**实现**：
```bash
#!/bin/bash
# scripts/upgrade.sh

set -e

CURRENT_VERSION=$(cat VERSION)
LATEST_VERSION=$(curl -s https://api.github.com/repos/kp-z/open_adventure/releases/latest | grep '"tag_name"' | sed -E 's/.*"v([^"]+)".*/\1/')

echo "当前版本: v$CURRENT_VERSION"
echo "最新版本: v$LATEST_VERSION"

if [ "$CURRENT_VERSION" = "$LATEST_VERSION" ]; then
    echo "✅ 已是最新版本"
    exit 0
fi

echo "🔄 开始升级..."

# 1. 停止服务
./stop.sh

# 2. 备份用户数据
echo "📦 备份用户数据..."
cp -r ~/.open_adventure ~/.open_adventure.backup.$(date +%Y%m%d_%H%M%S)

# 3. 下载新版本
echo "⬇️  下载新版本..."
PLATFORM=$(uname -s | tr '[:upper:]' '[:lower:]')
ARCH=$(uname -m)

if [ "$PLATFORM" = "darwin" ]; then
    PLATFORM="macos"
    ARCH="arm64"
elif [ "$PLATFORM" = "linux" ]; then
    PLATFORM="linux"
    ARCH="x86_64-compat"
fi

DOWNLOAD_URL="https://github.com/kp-z/open_adventure/releases/download/v${LATEST_VERSION}/open-adventure-v${LATEST_VERSION}-${PLATFORM}-${ARCH}.tar.gz"

curl -L -o /tmp/open-adventure-latest.tar.gz "$DOWNLOAD_URL"

# 4. 解压到临时目录
echo "📂 解压新版本..."
TEMP_DIR=$(mktemp -d)
tar -xzf /tmp/open-adventure-latest.tar.gz -C "$TEMP_DIR"

# 5. 替换二进制文件（保留用户数据）
echo "🔄 替换应用文件..."
CURRENT_DIR=$(pwd)

# 备份当前版本
mv "$CURRENT_DIR" "${CURRENT_DIR}.old"

# 移动新版本
mv "$TEMP_DIR/open-adventure" "$CURRENT_DIR"

# 6. 启动新版本
echo "🚀 启动新版本..."
cd "$CURRENT_DIR"
./start.sh --daemon

echo "✅ 升级完成！"
echo "📝 旧版本已保存到: ${CURRENT_DIR}.old"
echo "📝 数据备份已保存到: ~/.open_adventure.backup.*"
```

### 阶段 4：内置自动更新（v1.0.0 实现）

**功能**：
- 后台下载更新
- 自动替换二进制
- 无缝重启

**实现方案**：
使用独立的更新器进程（updater）：
1. 主应用检测到更新后，下载新版本到临时目录
2. 提示用户重启
3. 用户点击重启后，主应用启动 updater 进程
4. 主应用退出
5. updater 等待主应用完全退出
6. updater 替换二进制文件
7. updater 启动新版本主应用
8. updater 退出

## 数据迁移策略

### 用户数据目录结构
```
~/.open_adventure/
├── open_adventure.db          # SQLite 数据库
├── .env                        # 用户配置
├── frontend_dist/              # 前端静态资源
├── logs/                       # 日志文件
└── backups/                    # 自动备份
    ├── db_backup_20260317.db
    └── db_backup_20260316.db
```

### 自动迁移流程
1. **应用启动时**：
   - 检查数据库版本
   - 如果版本不匹配，自动备份数据库
   - 执行 Alembic 迁移
   - 迁移成功后删除备份
   - 迁移失败时恢复备份并提示用户

2. **迁移失败处理**：
   - 显示详细错误信息
   - 提供回滚选项
   - 提供手动迁移指南链接

## 版本兼容性策略

### 语义化版本规则
- **Major 版本**：可能包含不兼容的数据库变更
  - 需要提供迁移工具
  - Release Notes 中明确说明

- **Minor 版本**：向后兼容的新功能
  - 数据库 schema 可以自动迁移

- **Patch 版本**：Bug 修复
  - 不涉及数据库变更
  - 可以直接替换二进制

### 跨版本升级
- 支持跨 Minor 版本升级（如 v0.1.0 → v0.3.0）
- 不支持跨 Major 版本升级（如 v0.x → v1.x）
  - 需要先升级到最新的 v0.x
  - 然后再升级到 v1.0
