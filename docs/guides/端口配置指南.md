# Open Adventure 端口配置指南

**创建日期**: 2026-03-10
**更新日期**: 2026-03-10
**状态**: 已完成

## 概述

Open Adventure 默认使用端口 **38080** 作为后端服务端口。本文档说明如何配置端口以及二进制版本的限制。

## 默认端口变更说明

**⚠️ 重要变更**: 从 v1.3.6 开始，默认端口从 8000 改为 38080

**变更原因**:
- 8000 是常用端口，容易与其他服务冲突（Python HTTP 服务器、Django 开发服务器等）
- 38080 是冷门端口，减少端口冲突的可能性
- 易于记忆：38 = 3+8=11，080 = 80 的变体

**迁移指南**:
- 如果你之前使用默认端口 8000，升级后需要更新防火墙规则或反向代理配置
- 如果你显式指定了端口（`--port 8000` 或 `PORT=8000`），无需修改

## 打包版本（Linux/macOS 二进制）

### 命令行参数

打包后的二进制文件支持以下命令行参数：

```bash
./open-adventure --help
```

**可用参数**:
- `--port PORT` - 指定服务端口（默认: 38080）
- `--host HOST` - 指定监听地址（默认: 0.0.0.0）
- `--no-browser` - 不自动打开浏览器
- `-d, --daemon` - 后台运行模式

### 使用示例

#### 1. 指定端口运行

```bash
# 使用 38081 端口
./open-adventure --port 38081

# 使用 9000 端口
./open-adventure --port 9000
```

#### 2. 后台运行并指定端口

```bash
# 后台运行，使用 38081 端口
./open-adventure --daemon --port 38081
```

#### 3. 指定监听地址和端口

```bash
# 仅监听本地，使用 38081 端口
./open-adventure --host 127.0.0.1 --port 38081

# 监听所有网卡，使用 9000 端口
./open-adventure --host 0.0.0.0 --port 9000
```

#### 4. 不自动打开浏览器

```bash
# 启动服务但不打开浏览器
./open-adventure --port 38081 --no-browser
```

### 端口冲突处理

如果指定的端口已被占用，程序会自动检测并提示：

```
⚠️  端口 38080 已被占用
请使用 --port 参数指定其他端口，例如: ./open-adventure --port 38081
```

### 二进制版本的端口限制

**⚠️ 重要**: 二进制版本的前端在构建时已固定后端地址

这意味着：
- `--port` 参数只会修改后端监听端口
- 前端仍然会尝试连接到构建时配置的端口（默认 38080）
- 如果修改了后端端口，前端将无法连接到后端

**解决方案**:
1. **推荐**: 使用默认端口 38080，确保端口未被占用
2. **临时方案**: 使用反向代理（如 nginx）将请求转发到新端口
3. **重新构建**: 修改前端配置并重新构建二进制版本

### 后台运行模式

后台运行时，程序会：
1. 检查是否已有实例在运行
2. 创建 PID 文件：`~/.open_adventure/open_adventure.pid`
3. 输出日志到：`~/.open_adventure/open_adventure.log`

**查看后台服务状态**:
```bash
# 查看 PID
cat ~/.open_adventure/open_adventure.pid

# 查看日志
tail -f ~/.open_adventure/open_adventure.log

# 停止服务
kill $(cat ~/.open_adventure/open_adventure.pid)
```

## 开发环境（源码运行）

### 使用 start.sh 脚本

开发环境使用 `start.sh` 脚本启动，端口配置通过环境变量：

#### 1. 临时指定端口

```bash
# 指定后端端口
PORT=38081 ./start.sh

# 同时指定前后端端口
PORT=38081 FRONTEND_PORT=5174 ./start.sh
```

#### 2. 使用 .env 文件

创建或编辑 `backend/.env` 文件：

```bash
# backend/.env
PORT=38081
FRONTEND_PORT=5174
```

然后正常启动：
```bash
./start.sh
```

### 环境变量说明

- `PORT` - 后端服务端口（默认: 38080）
- `PREVENT_SLEEP` - macOS 防休眠模式（默认: false）

**注意**：前端已集成到后端，统一使用 38080 端口访问。

## 端口配置优先级

### 打包版本

1. 命令行参数 `--port`
2. 环境变量 `PORT`
3. 默认值 `38080`

### 开发环境

1. 环境变量 `PORT`
2. `.env` 文件中的 `PORT`
3. 默认值 `38080`

## 故障排查

### 问题 1: 端口被占用

**症状**: 启动时提示端口已被占用

**解决方案**:
```bash
# 查看占用端口的进程
lsof -i :38080

# 或使用其他端口
./open-adventure --port 38081
```

### 问题 2: 后台服务无法启动

**症状**: 后台模式启动失败

**解决方案**:
```bash
# 查看日志
cat ~/.open_adventure/open_adventure.log

# 检查是否已有实例运行
cat ~/.open_adventure/open_adventure.pid

# 清理旧的 PID 文件
rm ~/.open_adventure/open_adventure.pid
```

### 问题 3: 无法访问局域网地址

**症状**: 局域网其他设备无法访问

**解决方案**:
```bash
# 确保监听所有网卡
./open-adventure --host 0.0.0.0 --port 38080

# 检查防火墙设置
# macOS
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --add /path/to/open-adventure

# Linux (ufw)
sudo ufw allow 38080/tcp

# Linux (firewalld)
sudo firewall-cmd --permanent --add-port=38080/tcp
sudo firewall-cmd --reload
```

## 技术实现

### 端口检测逻辑

```python
def check_port_available(port: int) -> bool:
    """检查端口是否可用"""
    import socket
    try:
        with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
            s.bind(('0.0.0.0', port))
            return True
    except OSError:
        return False
```

### 命令行参数解析

```python
def parse_args():
    """解析命令行参数"""
    parser = argparse.ArgumentParser(description="Open Adventure")
    parser.add_argument("--port", type=int, default=None, help="服务端口 (默认: 38080)")
    parser.add_argument("--host", type=str, default="0.0.0.0", help="监听地址 (默认: 0.0.0.0)")
    parser.add_argument("--no-browser", action="store_true", help="不自动打开浏览器")
    parser.add_argument("-d", "--daemon", action="store_true", help="后台运行模式")
    return parser.parse_args()
```

## 总结

Open Adventure 提供了灵活的端口配置方式：
- **默认端口**: 38080（从 v1.3.6 开始，之前为 8000）
- **打包版本**: 使用 `--port` 参数（注意：前端地址在构建时固定）
- **开发环境**: 使用环境变量 `PORT`
- **端口冲突**: 自动检测并提示
- **二进制版本限制**: 前端连接地址在构建时固定，修改端口需要重新构建或使用反向代理

无论是开发、测试还是生产部署，都可以轻松配置端口。
